ClinicOS Conversations — Technical Architecture Document
Version: 1.0
Status: Design (pre-implementation)
Audience: Engineering, Product, Security
Scale target: 10,000+ clinics · 50M+ messages/month · 99.9% availability

Executive Summary
ClinicOS Conversations is a multi-tenant omnichannel patient communication platform embedded in ClinicOS. Each clinic operates in an isolated workspace with its own WhatsApp Business Account (WABA), conversation history, templates, campaigns, and automations.

This document defines the complete technical architecture. It extends the existing ClinicOS stack (Next.js 16, Supabase PostgreSQL, Vercel) with dedicated messaging infrastructure (Redis, BullMQ, background workers) required for enterprise scale.

1.4 Request Flow — Staff Sends a Message
1. Receptionist types message in ChatPanel
2. Server Action: sendMessageAction({ conversationId, content })
3. Conversation Service validates:
   - User has conversations:write permission
   - conversation.clinic_id === user.clinic_id
   - WhatsApp connection is active
   - 24-hour session window OR approved template required
4. Insert message row (status: queued)
5. Enqueue job: outbound-message { messageId }
6. Return optimistic UI update to client
7. Worker picks up job:
   a. Load clinic WhatsApp credentials from vault (decrypt)
   b. If session message: POST /messages (type: text)
   c. If outside 24h window: resolve template + POST template message
   d. Store external_message_id
   e. Update message status: sent
   f. Publish Realtime event on conversation channel
8. Meta webhook delivers status: delivered → read
9. Webhook Service enqueues status-update job
10. Worker updates message + message_status_history
11. Staff UI receives Realtime update (delivery ticks)
1.5 Event Flow — Patient Sends Inbound WhatsApp
1. Patient sends WhatsApp to clinic number
2. Meta POST → /api/webhooks/whatsapp
3. Webhook Service:
   a. Verify X-Hub-Signature-256
   b. Log raw payload → webhook_logs
   c. Extract metadata.phone_number_id
   d. Resolve clinic_id via whatsapp_connections
   e. Check idempotency key (wamid)
   f. Enqueue: inbound-message { payload }
   g. Return HTTP 200 immediately (<200ms)
4. Inbound Worker:
   a. Normalize phone, resolve/create patient
   b. Upsert conversation (channel: whatsapp)
   c. Insert message (direction: inbound)
   d. Increment unread_count
   e. Update conversation.last_message_at
   f. Publish Realtime: new message + inbox update
   g. Enqueue: ai-reply OR notify-staff (based on clinic settings)
5. AI Worker (if auto-reply enabled):
   a. Load conversation context + patient EMR summary
   b. Run agent with tool access (appointments, FAQ)
   c. If confidence high + no escalation trigger → send reply
   d. Else → set conversation.status = needs_human, notify staff
6. Notification Worker:
   a. Insert staff notification
   b. Supabase Realtime broadcast
   c. Optional email digest if unassigned >5min
2. Multi-Tenant Architecture
2.1 Clinic Isolation Model
Tenant = Clinic (clinics.id)

Every Conversations table includes clinic_id UUID NOT NULL REFERENCES clinics(id) ON DELETE CASCADE. No cross-clinic queries except Super Admin analytics (separate read path with audit).

Isolation layers (defense in depth):

Database: PostgreSQL RLS — clinic_id = my_profile_clinic_id()
Application: All service methods require clinicId parameter; assert on every read/write
Queue jobs: Every job payload includes clinicId; workers reject mismatches
Realtime: Supabase channels scoped to clinic:{clinicId}:conversations
Storage: Attachments at /{clinic_id}/conversations/{conversation_id}/
Meta API: Each clinic uses its own WABA token; no shared send identity
2.2 Meta Embedded Signup — Per-Clinic WhatsApp Connection
ClinicOS registers as a Meta Tech Provider (or Solution Partner). Each clinic connects its own WABA through Embedded Signup — a hosted Meta OAuth flow embedded in ClinicOS settings.

Prerequisites (platform level, one-time):

Meta Business Portfolio for ClinicOS
WhatsApp Business Platform app in Meta Developer Console
Tech Provider / Solution Partner approval
Embedded Signup configuration (redirect URI, permissions)
System User with whatsapp_business_management, whatsapp_business_messaging
Webhook subscription at app level
Clinic onboarding flow:

├─────────────────────────────────────────────────────────────┤

3. Database Design
3.1 Design Principles
Normalize conversations and messages — separate thread state from message content
Append-only status history — never overwrite delivery state; audit trail
Idempotency — external_message_id (wamid) unique constraint
Soft archive — conversations archived, not deleted (compliance)
Partition-ready — messages partitioned by created_at monthly at scale
Denormalize for inbox — conversations carries last_message_preview for fast list queries

7.5 Implementation Strategy
Phase 1 (MVP): Supabase Realtime postgres_changes on messages and conversations tables. Filter by clinic_id. Simple, no extra infra.

Phase 2 (Scale): Workers publish to Supabase Realtime Broadcast API directly (bypasses WAL replication latency). Reduces DB load at high message volume.

Phase 3 (10k+ clinics): Dedicated WebSocket server (Socket.io or Supabase Realtime dedicated cluster) with Redis pub/sub fanout. Supabase Realtime may hit limits at extreme scale.
